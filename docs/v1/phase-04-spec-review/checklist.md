# Spec Review Checklist

## Source Alignment

- [ ] The spec matches the original FD.
- [ ] The spec stays within its outline item.
- [ ] The spec does not silently add unrelated scope.
- [ ] Constraints from the FD are preserved.

## Behavior

- [ ] Expected behavior is clear.
- [ ] Current behavior is described where relevant.
- [ ] User flows are complete enough to understand the feature.
- [ ] Rules are specific and testable.
- [ ] Non-goals are explicit.

## States And Edge Cases

- [ ] Empty states are covered.
- [ ] Invalid input is covered.
- [ ] Permission denied behavior is covered where relevant.
- [ ] Conflict or duplicate behavior is covered where relevant.
- [ ] Failure or partial success behavior is covered where relevant.

## Cross-Spec Consistency

- [ ] Specs do not contradict each other.
- [ ] Shared terms mean the same thing across specs.
- [ ] Responsibilities do not overlap in confusing ways.
- [ ] Dependencies between specs are explicit.

## Acceptance Criteria

- [ ] Acceptance criteria are written in testable language.
- [ ] Acceptance criteria cover the main success path.
- [ ] Acceptance criteria cover important failure or edge paths.
- [ ] Acceptance criteria avoid implementation details.

- [ ] Approved
- [ ] Approved With Notes
- [ ] Needs Revision
